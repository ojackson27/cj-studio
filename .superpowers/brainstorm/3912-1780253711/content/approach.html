How should the 3D prism animation work?

The existing WebsiteAssembly component assembles a mock browser/website on scroll. We need to decide how the new glass prism animation fits with it.

A

Augment — Prisms float around the browser mock

Keep the existing browser assembly exactly as-is. Add 3–4 floating glass prism shapes that fall into position around the outside of the browser frame as you scroll. They act as decorative "atmosphere" — the prism logo shape, a sphere, a rectangle — orbiting the main UI demo.

Pros

  • Lowest risk — existing animation untouched
  • Fastest to build
  • Prisms reinforce brand without replacing content

Cons

  • Two animations competing for attention
  • Less dramatic/cinematic than the reference video
B

Replace — Full 3D prism assembly (as per JSON spec)

Completely replace the browser mock with the glass prism animation from the JSON. CSS glass cards fall from above, rotate in 3D, and land to form the website layout structure — nav, hero, service cards — but styled as iridescent glass prisms rather than a browser window.

Pros

  • Most cinematic — matches the reference video exactly
  • Unique, memorable scroll experience
  • Fully on-brand with the prism motif

Cons

  • Loses the existing browser-in-browser demo
  • More complex to build and test
  • 3D CSS has edge cases across browsers/devices
C

Two-phase — Prisms fall, then reveal the browser mock

First half of scroll: glass prisms fall and assemble (like the video). Second half: the prisms "dissolve" or part to reveal the existing browser mock assembling underneath. Two acts in one section.

Pros

  • Keeps both animations
  • Most dramatic narrative arc

Cons

  • 280vh scroll section may feel very long
  • Highest complexity — two animation systems to sync
  • Transition between acts is tricky to make smooth
