Three Ways to Structure the Workflow

Pick the architecture before we write any of it — each has a different shape when handed to a Claude agent

A

One Big Decision Tree

A single Markdown document — one long flow from "client enquiry received" all the way to Vercel deploy. Every branch lives inside it. You hand the whole thing to Claude as context.

Pros

  • One file to rule everything
  • Agent has full context always
  • Easy to share with clients or Josh

Cons

  • Gets very long (4 site types × 5 phases)
  • Hard to update one track without touching the whole doc
  • Agent reads irrelevant branches every time
B

Master Intake + 4 Track Playbooks Recommended

A short master doc handles client intake and classification. Once the site type is decided, a focused track-specific playbook takes over — its own tool list, design direction, ARCHITECTURE.md template, and brief format. Claude only reads the relevant playbook.

Pros

  • Each playbook is lean and focused
  • Agent gets only relevant context — faster, cheaper
  • Easy to update one track without touching others
  • Scales cleanly as you add new site types

Cons

  • Multiple files to maintain
  • Need a clear handoff point between master and track
File structure: WORKFLOW.md (master intake) → tracks/hospitality.md · tracks/trades.md · tracks/professional.md · tracks/creative.md
C

Claude Prompt Template System

Instead of a reference doc, build it as a set of structured Claude prompts. A "classifier prompt" takes the client brief and outputs the site type + a generated ARCHITECTURE.md. You paste that output into the existing build workflow.

Pros

  • Most automatable — closest to full agent hand-off
  • Claude generates the ARCHITECTURE.md, not you
  • Very token-efficient at runtime

Cons

  • Harder to read/review as a human
  • Quality depends on prompt engineering
  • Less transparent — harder to audit what Claude decided
